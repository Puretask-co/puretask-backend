  stripe_transfer_id TEXT NULL,

);

CREATE INDEX IF NOT EXISTS idx_payouts_requested_at ON payouts (requested_at DESC);
3.3 Cleaner Ledger — append‑only
-- 003_create_cleaner_ledger.sql
CREATE TYPE ledger_type AS ENUM ('earning','payout','fee','adjustment','chargeback');

CREATE TABLE IF NOT EXISTS cleaner_ledger (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  cleaner_id UUID NOT NULL REFERENCES users(id),
  type ledger_type NOT NULL,
  amount_cents INT NOT NULL, -- signed; payouts/fees are negative
  ref_type TEXT NOT NULL,    -- e.g. 'job','payout','refund','chargeback'
  ref_id TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_cleaner_ledger_cleaner_created ON cleaner_ledger (cleaner_id, created_at DESC);
4) State Machine (authoritative rules)
Statuses: pending → processing → completed|failed and rejected (terminal).

Only admin action sets pending → processing or pending → rejected.

Only Stripe webhook may set processing → completed|failed.

Every transition writes an audit stamp (implicit via timestamps + event source in API logs) and, when relevant, a ledger entry.

Idempotency

Each payout has a unique idempotency_key used across API + n8n + Stripe idempotency_key header. Retries must not duplicate transfers.

5) Business Rules
KYC gating: Payouts only if stripe_connect_id != '' and connect_requirements_ok = TRUE.

Minimum payout: MIN_PAYOUT_CENTS (default 2000).

First‑payout cooldown: FIRST_PAYOUT_COOLDOWN_HRS (e.g., 72h since first earning) to reduce fraud.

Negative balance handling: If refunds/chargebacks push the ledger negative, accumulate and auto‑recoup from next payout.

Instant payout fee: INSTANT_FEE_BPS (e.g., 150 = 1.5%) + Stripe rail cost; stored in platform_fee_cents.

Caps: MAX_PAYOUT_PER_DAY_CENTS and per‑week guardrails.

Source of truth: Ledger sums compute available balance; payouts rows reflect requests/attempts, not balances.

Update existing data

Delete data

🏗️ 2. What PostgreSQL Is (The Database System)
PostgreSQL (Postgres) is a database engine — a program that stores, organizes, and retrieves your data using SQL.

Think of:

SQL = the language you speak

PostgreSQL = the machine that understands it and saves the data

There are other database engines (like MySQL, SQLite, MongoDB), but Postgres is special because it’s:

Reliable (handles millions of records safely)

Open-source (free and community-built)

Feature-rich (supports transactions, JSON, geolocation, etc.)

Perfect for marketplaces (like Uber, Airbnb, and PureTask)

wats idempotency

Excellent question — idempotency is one of the most important but least understood concepts in building systems like PureTask, especially when handling payments, webhooks, and automations.

Let’s break it down in plain English — and then look at how PureTask uses it in payouts, refunds, and automations.

🧠 1. The Simple Definition
Idempotency means:

Doing the same thing twice should have the same result as doing it once.

Or said another way:

“If I accidentally press the button twice, nothing bad should happen.”

💳 Example (real life)
Imagine you’re at checkout and you click “Pay Now” twice.

Without idempotency → 💥 you get charged twice.
With idempotency → ✅ the system realizes “Hey, I already did that exact charge” and ignores the duplicate.

That’s idempotency.

⚙️ 2. Why It Matters for PureTask
PureTask handles money movement, credits, and Stripe payouts, so the system must be idempotent — or else it could double-charge clients or double-pay cleaners.

Examples in PureTask where idempotency is essential:

Client buys credits → must create one transaction, even if the API retries.

Cleaner payout → must send one Stripe transfer, even if the admin clicks “Approve” twice.

Webhooks → Stripe might resend the same “payout.succeeded” event multiple times; we only want to mark it completed once.

🧩 3. The Core Idea (How Systems Enforce It)
Every time you perform an action that could be repeated (like a payout), you give it a unique ID — called an idempotency key.

When the system sees that same key again, it says:

“Oh, I’ve already processed this — skip it.”

🧱 Example: Payout in PureTask
Admin approves payout for Cleaner ID cln_123.

Backend creates an idempotency_key like:

76f1e7b0-5e7a-4a51-8d12-7f2b2f7c9a10
System saves it in the payouts table:

INSERT INTO payouts (cleaner_id, amount_cents, idempotency_key, status)
VALUES ('cln_123', 4500, '76f1e7b0...', 'processing');
Backend calls Stripe API with that same key:

POST /v1/transfers
Idempotency-Key: 76f1e7b0-5e7a-4a51-8d12-7f2b2f7c9a10
If Stripe or your network glitches and retries, Stripe recognizes the key → returns the same transfer ID instead of making a new payout.

Result:
✅ One payout created, even if retried multiple times.

🔄 4. Where PureTask Uses Idempotency
System	What it prevents	Example
Payments (Credits)	Double-charging clients	Two identical charge.succeeded events won’t add credits twice
Payouts (Stripe)	Double-paying cleaners	Two API calls with same key → Stripe returns same transfer
Refunds	Double-refunding clients	Same refund request → one refund ID reused
Webhooks	Duplicate event processing	Stripe resends same webhook → PureTask checks if already handled
n8n automations	Re-run safety	Workflow checks key before running new step
🧰 5. How You Implement It in Code
Step 1 — Generate a Unique Key
Every request gets a unique UUID or hash.

import { v4 as uuidv4 } from 'uuid';
const key = uuidv4();
Step 2 — Store It in Your Database
Keep it in the record:

INSERT INTO payouts (cleaner_id, amount_cents, idempotency_key)
VALUES ('cln_123', 4500, '76f1e7b0-...');
Step 3 — Check Before Running
When you run your automation or API call, do:

SELECT * FROM payouts WHERE idempotency_key = '76f1e7b0-...';
If it already exists → skip.

🧩 6. Real Example in Stripe
Stripe uses idempotency keys natively — you just send them in your API headers:

POST https://api.stripe.com/v1/transfers
Idempotency-Key: 76f1e7b0-5e7a-4a51-8d12-7f2b2f7c9a10
If you call this twice:

Without a key → you get two transfers (bad)

With a key → you get the same transfer object (good)

🧠 7. PureTask’s Rule of Thumb
Whenever something involves:

Money

State changes (e.g., status: pending → completed)

Webhook events

Automations triggered by external APIs

➡️ Always use an idempotency key.

🧩 8. In Simple Terms
Without Idempotency	With Idempotency
You click “Pay” twice → charged twice	You click “Pay” twice → one charge
Automation retries payout → two transfers	Automation retries payout → one transfer
Stripe webhook fires twice → duplicate DB updates	Stripe webhook fires twice → ignored duplicate
💬 9. Analogy
Think of idempotency like a ticket number on an order.

If your restaurant system gets two identical orders but sees the same ticket number, it says:

“Already made that burger.”

If it doesn’t use ticket numbers, it makes two burgers 🍔🍔 and charges twice.
