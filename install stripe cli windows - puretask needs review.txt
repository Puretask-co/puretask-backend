



wat is cloudflare?

Cloudflare is a security + performance service that sits in front of your website or app.

Think of it as a shield + accelerator between users and your servers.

Simple explanation (plain English)
When someone visits your app or website:

User → Cloudflare → Your server

Cloudflare:

protects you from attacks

makes your site load faster

keeps your site online if something goes wrong

What Cloudflare actually does (the important parts)
1. Security (huge)
Cloudflare protects your app from:

hackers

bots

fake traffic

DDoS attacks (when someone tries to crash your site with traffic)

For PureTask, this is critical because you have:

logins

payments

Stripe webhooks

admin dashboards

Cloudflare blocks bad traffic before it ever reaches n8n, your API, or your database.

2. Speed (CDN)
Cloudflare has servers all over the world.

It:

caches images, scripts, and pages

serves them from the closest location to the user

Result:

faster load times

smoother UX

better SEO

Example:
A user in New York doesn’t have to wait for your West Coast server.

3. DNS (domain control)
Cloudflare usually becomes your DNS provider.

That means:

you manage puretask.com

you point it to:

your frontend

your API

n8n

Stripe webhooks

Cloudflare = control center for where traffic goes.

4. Free HTTPS / SSL
Cloudflare gives you:

HTTPS

SSL certificates

automatic renewal

No setup pain.

This is required for:

Stripe

logins

mobile apps

browser trust

5. Firewall rules
You can say things like:

“Block traffic from these countries”

“Only allow Stripe webhooks to hit this endpoint”

“Rate-limit login attempts”

For PureTask, this is 🔥 because you can:

lock down /stripe/webhook

protect /admin

stop brute-force attacks

What Cloudflare is NOT
❌ It is not your backend
❌ It does not store your database
❌ It does not replace n8n, Base44, or Stripe

It protects and optimizes them.

How Cloudflare fits into PureTask (specifically)
For PureTask, Cloudflare should sit in front of:

app.puretask.com (frontend)

api.puretask.com (backend)

n8n.puretask.com (automations + webhooks)

Flow example:
Client books a cleaning →
Stripe sends webhook →
Cloudflare checks it →
Passes it to n8n →
n8n updates database + payouts

If Cloudflare sees something sketchy → blocked.

Do you need Cloudflare?
Yes. 100%.

Any marketplace with:

payments

…should be using Cloudflare.

It’s:

free to start

used by Stripe, Shopify, Discord, Uber-scale apps

is there software that takes the backend data and creates the front end?

Yes — there is software that turns backend data into a frontend, but it’s important to understand the different levels, because they’re not all equal.

I’ll explain this clearly, then tell you what actually makes sense for PureTask (not theory).

Short answer (plain English)
There is no magic tool that perfectly builds a custom consumer app UI from raw backend logic.

But there ARE tools that:

auto-generate admin dashboards

auto-render CRUD screens

bind databases/APIs to UI components

dramatically reduce frontend work

Most real companies use a hybrid approach.

The 4 categories (from weakest → strongest)
1️⃣ Admin / Internal Tools (YES, very real)
These tools read your backend data and instantly generate a frontend.

They are GREAT for:

admin panels

ops dashboards

internal tools

back-office views

Examples
Retool

Supabase Studio

Hasura Console

Appsmith

Forest Admin

What they do
Connect to your database or API

Auto-generate tables, forms, filters

Let you edit records visually

What they do NOT do
❌ Not good for consumer-facing apps
❌ Not good for polished UX
❌ Not mobile-app friendly

Verdict
✅ You should absolutely use this for:

admin panel

cleaner moderation

payouts review

disputes

metrics

2️⃣ Low-Code / No-Code App Builders (PARTIAL)
These tools connect to backend data and let you visually build screens.

Examples
Bubble

FlutterFlow

WeWeb

Framer + CMS

Softr

What they do
Bind backend data to UI components

Generate real frontends

Handle auth, forms, lists, detail views

Limits
❌ Hard to scale long-term
❌ Complex logic gets messy
❌ Performance limits
❌ Vendor lock-in

Verdict
⚠️ Good for MVPs
⚠️ Not ideal for a serious marketplace long-term

3️⃣ Headless Backend + UI Bindings (BEST PRACTICE)
This is what real production apps use.

Your backend exposes:

structured APIs

schemas

events

Your frontend:

consumes those APIs

renders UI intentionally

But tooling helps bridge the gap.

Examples
Supabase + React

Firebase + Flutter

PostgREST

tRPC

GraphQL

Stripe Elements (auto-UI from backend config)

This gives you:

backend-driven logic

frontend-controlled UX

Verdict
✅ This is what PureTask should do

4️⃣ True “Backend → UI Auto-Generation” (REALITY CHECK)
People want:

“I define logic and data, and the app UI appears.”

That only works well for:

internal tools

CRUD apps

enterprise dashboards

It does not work well for:

how can i download my project from github

Here are the 4 correct ways to download a project from GitHub, from easiest → most “developer-correct”. Pick the one that matches your comfort level.

✅ Method 1: Download ZIP (fastest, no tools)
Best if you just want the files on your computer.

Go to your GitHub repository in the browser

Click the green “Code” button

Click “Download ZIP”

Unzip the file on your computer

That’s it.

⚠️ Notes:

This does not include git history

You can’t easily push changes back unless you re-init git

✅ Method 2: Clone with Git (best practice)
Best if you plan to edit, version, and push changes back.

Step 1: Copy repo URL
Click Code

Copy the HTTPS URL
(example: https://github.com/username/project.git)

Step 2: Open terminal (or Command Prompt)
Run:

git clone https://github.com/username/project.git
This creates a local folder with:

full project

full commit history

ability to push/pull updates

📌 You’ll need Git installed
Check with:

git --version
✅ Method 3: GitHub Desktop (no terminal)
Best if you don’t want to use command line.

Download GitHub Desktop

Sign in to your GitHub account

Go to your repo in the browser

Click Code → Open with GitHub Desktop

Choose a local folder

Click Clone

Now you have:

local project

visual UI for commits & pushes

✅ Method 4: Download a specific branch or folder
Download a specific branch
Switch branches in GitHub UI

Click Code → Download ZIP

Download only one folder (advanced)
GitHub doesn’t support this natively, but tools exist:

git sparse-checkout

third-party downloaders

(Only worth it for large repos.)

Which should you use?
For PureTask, I recommend:

If you’re:
reviewing code

moving it to another environment

backing it up

➡ Method 1 (ZIP)

If you’re:
actively building

collaborating

deploying

➡ Method 2 (git clone) or Method 3 (GitHub Desktop)

Common mistakes to avoid
❌ Downloading ZIP and thinking it’s “connected” to GitHub
❌ Editing files without git if you plan to deploy
❌ Losing environment variables (they are NOT in the repo)

how can we find out wat screen we should have for the customer interactions

This is a product-design question, not a design-tools question — and that’s good.
There’s a correct way to figure out what screens you need, and it’s not “guessing” or copying other apps.

I’ll walk you through the method, then apply it directly to PureTask customers.

The core principle (lock this in)
Screens are not features.
Screens are answers to user questions at specific moments in time.

So the way we find screens is by asking:

What is the customer trying to do right now?

What uncertainty do they have?

What decision must they make next?

Step 1: Start with the CUSTOMER JOURNEY (not screens)
For PureTask customers, the journey is:

“Can I trust this?”

“How does this work?”

“Who is coming to my home?”

“Is the cleaner actually on the way?”

“Did they do a good job?”

“Am I being charged fairly?”

“What if something went wrong?”

Each of those forces at least one screen.

Step 2: Convert questions → moments → screens
1️⃣ “Can I trust this?”
Moment: First time opening the app

👉 Screen needed:

Landing / Home screen
Purpose:

Explain approve-before-pay

Show trust signals (photos, GPS, verification)

One clear CTA: Book a cleaning

2️⃣ “How does this work?”
Moment: Before committing money

👉 Screen needed:

How it Works screen (or section)
Purpose:

3–4 steps max

Reduce anxiety

Explain credits + approval simply

If you skip this, people hesitate.

3️⃣ “Who is coming to my home?”
Moment: Choosing a cleaner

👉 Screens needed:

Cleaner List screen

Cleaner Profile screen

Purpose:

Face + name

Reliability score

Reviews

Price range

Verification badges

Trust is visual here.

4️⃣ “What am I paying & what’s being held?”
Moment: Booking

👉 Screen needed:

Booking / Checkout screen

Purpose:

Show:

estimated hours

credits held

buffer explanation

Reinforce: You only pay for time worked

This is where most marketplaces lose users.

5️⃣ “Is the cleaner actually on the way?”
Moment: Day of service

👉 Screen needed:

Job In Progress screen

Purpose:

Status timeline:

accepted

on the way

checked in

in progress

GPS indicator

Calm reassurance

This screen kills anxiety.

6️⃣ “Did they do a good job?”
Moment: Job completed

👉 Screen needed:

Job Completion / Review screen

Purpose:

Before & after photos

Time worked

Final charge

Approve / dispute buttons

This is your core differentiator.

7️⃣ “Am I being charged fairly?”
Moment: Approval

👉 Screen needed:

Approval & Payment Summary screen

Purpose:

Final credits charged

Credits refunded (if any)

Clear confirmation

This builds long-term trust.

8️⃣ “What if something went wrong?”
Moment: Problem or doubt

👉 Screens needed:

Dispute screen

Support / Help screen

Purpose:

Show they’re protected

Simple options, not legal text

Even if rarely used, this increases conversion.

Step 3: Group screens into a MINIMAL customer set
For PureTask customers, the minimum complete set is:

Core customer screens (must-have)
Home / Landing

How It Works

Cleaner List

Cleaner Profile

Booking / Checkout

Job In Progress

Job Completed / Review

Approval / Payment Summary

Wallet / Credits

Support / Dispute

That’s it.
Anything beyond this is secondary.

Step 4: Validate screens with the “What would break if we remove this?” test
Ask for each screen:

“If this didn’t exist, would the customer feel confused, unsafe, or blocked?”

If yes → required
If no → optional or merged

Example:

Notifications screen? Optional (can be settings)

Wallet screen? Required (money clarity)

Profile settings? Optional early on

Step 5: Tie screens to backend STATES (this is where you win)
Each screen exists because the backend has a state.

Example:

Job status = in_progress → Job In Progress screen

Job status = awaiting_approval → Approval screen

Credits on hold → Wallet explanation

This is how backend drives screens, but design decides how they look.

The big insight (this matters)
You don’t design screens first.
You design moments of uncertainty → then create screens to resolve them.

That’s why good apps feel calm and bad ones feel stressful.

import { base44 } from "@/base44.ts";

export async function idempotencyGate(opts: {
  key: string | null;
  userId: string;
  endpoint: string;
  requestHash: string;
}) {
  const { key, userId, endpoint, requestHash } = opts;

  if (!key) {
    return { ok: false, status: 400, error: { code: "IDEMPOTENCY_REQUIRED", message: "Missing Idempotency-Key" } };
  }

  const existing = await base44.entities.idempotency_keys.get(key).catch(() => null);

  if (existing) {
    if (existing.endpoint !== endpoint || existing.user_id !== userId) {
      return { ok: false, status: 409, error: { code: "IDEMPOTENCY_CONFLICT", message: "Idempotency key already used for another request" } };
    }
    if (existing.request_hash !== requestHash) {
      return { ok: false, status: 409, error: { code: "IDEMPOTENCY_CONFLICT", message: "Idempotency key used with different payload" } };
    }
    if (existing.status === "completed" && existing.response_json) {
      return { ok: true, replay: true, response: existing.response_json };
    }
    // If processing/failed, you can allow retry logic here.
  }

  await base44.entities.idempotency_keys.create({
    idempotency_key: key,
    user_id: userId,
    endpoint,
    request_hash: requestHash,
    status: "processing",
    response_json: null,
  });

  return { ok: true, replay: false };
}

export async function idempotencyComplete(key: string, responseJson: unknown) {
  await base44.entities.idempotency_keys.update(key, {
    status: "completed",
    response_json: responseJson,
  });
}

export async function idempotencyFail(key: string, errorJson: unknown) {
  await base44.entities.idempotency_keys.update(key, {
    status: "failed",
    response_json: errorJson,
  });
}

export async function emitEvent(name: string, data: any) {
  const event = {
    event_id: crypto.randomUUID(),
    occurred_at: new Date().toISOString(),
    name,
    actor: data.actor,
    job_id: data.job_id ?? null,
    idempotency_key: data.idempotency_key ?? null,
    payload: data.payload ?? {},
  };

  // internal event bus
  await fetch(`${Deno.env.get("API_BASE_URL")}/events`, {
    method: "POST",
    headers: { "Content-Type": "application/json", "X-Internal-Token": Deno.env.get("INTERNAL_TOKEN")! },
    body: JSON.stringify(event),
  });

  // fanout to n8n (optional per event)
  if (data.fanoutToN8n) {
    await fetch(`${Deno.env.get("API_BASE_URL")}/n8n/events`, {
      method: "POST",
      headers: { "Content-Type": "application/json", "X-Internal-Token": Deno.env.get("INTERNAL_TOKEN")! },
      body: JSON.stringify(event),
    });
  }

  return event;
}

export default async function handler(req: Request) {
  const body = await req.json();
  const actor = await getActorFromCookie(req); // your auth helper
  const idKey = req.headers.get("Idempotency-Key");

  const requestHash = await hashJson({ body, actorId: actor.id });
  const gate = await idempotencyGate({ key: idKey, userId: actor.id, endpoint: "POST /jobs", requestHash });
  if (!gate.ok) return json(gate.error, gate.status);
  if (gate.replay) return json(gate.response, 200);

  try {
    // 1) validations
    assert(actor.role === "client", "FORBIDDEN");
    assert(new Date(body.scheduled_start_at) < new Date(body.scheduled_end_at), "VALIDATION_ERROR");

    // 2) compute available credits
    const available = await computeAvailableCredits(actor.id);

    assert(available >= body.credit_amount, "INSUFFICIENT_CREDITS");

    // 3) create job
    const job = await base44.entities.jobs.create({
      client_id: actor.id,
      status: "requested",
      scheduled_start_at: body.scheduled_start_at,
      scheduled_end_at: body.scheduled_end_at,
      address: body.address,
      latitude: body.latitude,
      longitude: body.longitude,
      credit_amount: body.credit_amount,
      client_notes: body.client_notes ?? null,
    });

    // 4) job event
    await base44.entities.job_events.create({
      job_id: job.id,
      type: "job_created",
      payload: {},
    });

    // 5) hold credits
    await base44.entities.credit_ledger.create({
      user_id: actor.id,
      job_id: job.id,
      delta_credits: -body.credit_amount,
      reason: "hold_for_job",
    });

    const response = { job, credits: { held: body.credit_amount } };

    await emitEvent("job.created", {
      actor: { type: "client", id: actor.id },
      job_id: job.id,
      idempotency_key: idKey,
      payload: { credit_amount: body.credit_amount },
      fanoutToN8n: false,
    });

    await emitEvent("credits.hold.created", {
      actor: { type: "client", id: actor.id },
      job_id: job.id,
      idempotency_key: idKey,
      payload: { delta_credits: -body.credit_amount, reason: "hold_for_job" },
      fanoutToN8n: false,
    });

    await idempotencyComplete(idKey!, response);
    return json(response, 200);
  } catch (e) {
    await idempotencyFail(idKey!, { error: String(e) }).catch(() => {});
    return json({ error: { code: "INTERNAL_ERROR", message: "Failed to create job" } }, 500);
  }
}

import { base44 } from "@/base44.ts";

type Role = "ops" | "finance" | "trust_safety" | "super_admin";
type ActorType = "admin" | "system";

export function json(body: unknown, status = 200) {
  return new Response(JSON.stringify(body), {
    status,
    headers: { "Content-Type": "application/json" },
  });
}

export function assert(cond: unknown, code: string, message?: string): asserts cond {
  if (!cond) throw Object.assign(new Error(message ?? code), { code });
}

export async function getAdminActor(req: Request) {
  // Implement using your session/cookie auth context.
  // Base44 usually gives you a current user context; if not, parse cookie and verify.
  const user = (req as any).user ?? null; // replace
  assert(user, "UNAUTHORIZED", "Not logged in");
  assert(user.role && user.role.startsWith("admin"), "FORBIDDEN", "Not an admin");
  return {
    id: user.id as string,
    role: user.admin_role as Role, // ops/finance/trust_safety/super_admin
    type: "admin" as ActorType,
  };
}

export function requireRole(actor: { role: Role }, allowed: Role[]) {
  assert(allowed.includes(actor.role), "FORBIDDEN", "Insufficient admin role");
}

/** Hash request payload for idempotency comparisons */
export async function hashJson(input: unknown): Promise<string> {
  const bytes = new TextEncoder().encode(JSON.stringify(input));
  const digest = await crypto.subtle.digest("SHA-256", bytes);
  return Array.from(new Uint8Array(digest)).map((b) => b.toString(16).padStart(2, "0")).join("");
}

// ---- Idempotency (admin actions) ----
export async function idempotencyGate(opts: {
  key: string | null;
  adminId: string;
  endpoint: string;
  requestHash: string;
}) {
  const { key, adminId, endpoint, requestHash } = opts;
  assert(key, "IDEMPOTENCY_REQUIRED", "Missing Idempotency-Key (action_id)");

  const existing = await base44.entities.idempotency_keys.get(key!).catch(() => null);
  if (existing) {
    if (existing.endpoint !== endpoint || existing.user_id !== adminId) {
      throw Object.assign(new Error("Idempotency key conflict"), { code: "IDEMPOTENCY_CONFLICT" });
    }
    if (existing.request_hash !== requestHash) {
      throw Object.assign(new Error("Idempotency key reused with different payload"), { code: "IDEMPOTENCY_CONFLICT" });
    }
    if (existing.status === "completed" && existing.response_json) {
      return { replay: true, response: existing.response_json };
    }
  }

  await base44.entities.idempotency_keys.create({
    idempotency_key: key!,
    user_id: adminId,
    endpoint,
    request_hash: requestHash,
    status: "processing",
    response_json: null,
  });

  return { replay: false, response: null };
}

export async function idempotencyComplete(key: string, responseJson: unknown) {
  await base44.entities.idempotency_keys.update(key, {
    status: "completed",
    response_json: responseJson,
  });
}
export async function idempotencyFail(key: string, errorJson: unknown) {
  await base44.entities.idempotency_keys.update(key, {
    status: "failed",
    response_json: errorJson,
  });
}

// ---- Audit log (mandatory) ----
export async function auditLog(opts: {
  action_id: string;
  admin_id: string;
  action: string;
  target_type: "job" | "dispute" | "user" | "ledger" | "config" | "stripe" | "notification";
  target_id: string;
  payload: Record<string, unknown>;
}) {
  await base44.entities.admin_audit_log.create({
    id: opts.action_id,
    admin_id: opts.admin_id,
    action: opts.action,
    target_type: opts.target_type,
    target_id: opts.target_id,
    payload: opts.payload,

  });
}

// ---- Canonical event emitter ----
export async function emitEvent(name: string, data: {
  actor: { type: "admin" | "system"; id: string };
  job_id?: string | null;
  idempotency_key?: string | null;
  payload?: Record<string, unknown>;
  fanoutToN8n?: boolean;
}) {
  const event = {
    event_id: crypto.randomUUID(),
    occurred_at: new Date().toISOString(),
    name,
    actor: data.actor,
    job_id: data.job_id ?? null,
    idempotency_key: data.idempotency_key ?? null,
    payload: data.payload ?? {},
  };

  // Store in Neon for unified tracking (Deliverable “track everything”)
  await base44.entities.observability_events.create({
    event_id: event.event_id,
    occurred_at: event.occurred_at,
    name: event.name,
    actor_type: event.actor.type,
    actor_id: event.actor.id,
    job_id: event.job_id,
    idempotency_key: event.idempotency_key,
    payload: event.payload,
    source: "base44",
  });

  // Optional: also post to your internal API bridges if you want
  // await fetch(`${Deno.env.get("API_BASE_URL")}/events`, { ... })

  if (data.fanoutToN8n) {
    await fetch(Deno.env.get("N8N_EVENTS_URL")!, {
      method: "POST",
      headers: { "Content-Type": "application/json", "X-Signature": Deno.env.get("N8N_HMAC")! },
      body: JSON.stringify(event),
    });
  }

  return event;
}

export function toErrorResponse(err: any) {
  const code = err?.code ?? "INTERNAL_ERROR";
  const status =
    code === "UNAUTHORIZED" ? 401 :
    code === "FORBIDDEN" ? 403 :
    code === "NOT_FOUND" ? 404 :
    code === "VALIDATION_ERROR" ? 400 :
    code === "CSRF_REQUIRED" ? 403 :
    code === "IDEMPOTENCY_REQUIRED" ? 400 :
    code === "IDEMPOTENCY_CONFLICT" ? 409 :
    code === "INVALID_STATE" ? 409 :
    500;

  return json({ error: { code, message: err?.message ?? "Error" } }, status);
}

import { base44 } from "@/base44.ts";
import {
  json, toErrorResponse, getAdminActor, requireRole,
  idempotencyGate, idempotencyComplete, idempotencyFail,
  auditLog, emitEvent, hashJson, assert
} from "./_shared.ts";

export default async function handler(req: Request) {
  const actor = await getAdminActor(req);
  requireRole(actor, ["ops", "trust_safety", "super_admin"]);

  const body = await req.json(); // { action_id, job_id, reason }
  const actionId = body.action_id || req.headers.get("Idempotency-Key");
  const endpoint = "admin.job.cancel";
  const requestHash = await hashJson({ body, actor: actor.id });

  try {
    const gate = await idempotencyGate({ key: actionId, adminId: actor.id, endpoint, requestHash });
    if (gate.replay) return json(gate.response, 200);

    assert(body.job_id, "VALIDATION_ERROR", "Missing job_id");
    assert(body.reason, "VALIDATION_ERROR", "Missing reason");

    const job = await base44.entities.jobs.get(body.job_id).catch(() => null);
    assert(job, "NOT_FOUND", "Job not found");

    // Guard: only cancel pre-start for V1
    const allowed = ["requested", "accepted", "on_my_way"];
    assert(allowed.includes(job.status), "INVALID_STATE", `Cannot cancel job in status ${job.status}`);

    // Update job + event
    await base44.entities.jobs.update(job.id, { status: "cancelled" });
    await base44.entities.job_events.create({
      job_id: job.id,
      type: "job_cancelled",
      payload: { reason: body.reason, by: "admin", admin_id: actor.id },
    });

    // Release hold (ledger reversal). We assume HOLD exists; if not, still succeed.
    // Strategy: Insert release entry (append-only).
    await base44.entities.credit_ledger.create({
      user_id: job.client_id,
      job_id: job.id,
      delta_credits: Math.abs(job.credit_amount),
      reason: "hold_released_on_cancel",
    }).catch(() => null);

    await auditLog({
      action_id: actionId,
      admin_id: actor.id,
      action: "admin.job.cancel",
      target_type: "job",
      target_id: job.id,
      payload: { reason: body.reason },
    });

    await emitEvent("job.cancelled", {
      actor: { type: "admin", id: actor.id },
      job_id: job.id,
      idempotency_key: actionId,
      payload: { reason: body.reason },
      fanoutToN8n: true, // so n8n can notify client/cleaner
    });

    await emitEvent("credits.hold.released", {
      actor: { type: "admin", id: actor.id },
      job_id: job.id,
      idempotency_key: actionId,
      payload: { user_id: job.client_id, delta_credits: Math.abs(job.credit_amount), reason: "hold_released_on_cancel" },
      fanoutToN8n: false,
    });

    const response = { ok: true, job_id: job.id, status: "cancelled" };
    await idempotencyComplete(actionId, response);
    return json(response, 200);
  } catch (err) {
    if (actionId) await idempotencyFail(actionId, { error: String(err) }).catch(() => {});
    return toErrorResponse(err);
  }
}

-- =========================================
-- PureTask Ops / Admin / Observability Tables
-- =========================================

BEGIN;

-- 1) Idempotency keys (critical writes + admin actions)
CREATE TABLE IF NOT EXISTS idempotency_keys (
  idempotency_key TEXT PRIMARY KEY,
  user_id         TEXT NOT NULL,
  endpoint        TEXT NOT NULL,         -- e.g. "POST /jobs" or "admin.job.cancel"
  request_hash    TEXT NOT NULL,         -- sha256 of canonical request payload
  status          TEXT NOT NULL CHECK (status IN ('processing','completed','failed')),
  response_json   JSONB,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_idempotency_keys_user_endpoint_created
  ON idempotency_keys (user_id, endpoint, created_at DESC);

-- Auto-update updated_at
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_idempotency_keys_updated_at ON idempotency_keys;
CREATE TRIGGER trg_idempotency_keys_updated_at
BEFORE UPDATE ON idempotency_keys
FOR EACH ROW EXECUTE FUNCTION set_updated_at();


-- 2) Admin audit log (immutable)
CREATE TABLE IF NOT EXISTS admin_audit_log (
  id          UUID PRIMARY KEY,          -- use action_id / Idempotency-Key uuid
  admin_id    TEXT NOT NULL,
  action      TEXT NOT NULL,             -- e.g. "admin.job.cancel"
  target_type TEXT NOT NULL CHECK (target_type IN ('job','dispute','user','ledger','config','stripe','notification')),
  target_id   TEXT NOT NULL,
  payload     JSONB NOT NULL DEFAULT '{}'::jsonb,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_admin_audit_log_created
  ON admin_audit_log (created_at DESC);

CREATE INDEX IF NOT EXISTS idx_admin_audit_log_admin_created
  ON admin_audit_log (admin_id, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_admin_audit_log_target
  ON admin_audit_log (target_type, target_id, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_admin_audit_log_action
  ON admin_audit_log (action, created_at DESC);


-- 3) System config (editable by Super Admin; everything else reads)
CREATE TABLE IF NOT EXISTS system_config (
  key         TEXT PRIMARY KEY,
  value       JSONB NOT NULL,
  description TEXT,
  updated_by  TEXT,
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

DROP TRIGGER IF EXISTS trg_system_config_updated_at ON system_config;
CREATE TRIGGER trg_system_config_updated_at
BEFORE UPDATE ON system_config
FOR EACH ROW EXECUTE FUNCTION set_updated_at();
